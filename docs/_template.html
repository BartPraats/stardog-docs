Introduction

PelletServer is a framework that's optimized for building integration and analysis applications, primarily using Semantic Web technologies—including SPARQL, RDF, and OWL. PelletServer provides programmatic access via a simple REST interface. In most cases, accessing PelletServer's services is as simple as issuing an HTTP GET. While most of PelletServer's capabilities are implemented in Java—which means that programming with them requires programming in Java—the REST interface makes them universally accessible.

Focus on the Data: PelletServer

PelletServer is a backend system that provides data and computational services for frontend or user-facing system components; it contains a simple user interface for administrative functions, but its not intended to be directly user-facing. Because PelletServer is based entirely on REST, any competent, modern web app framework will integrate cleanly with it: Rails, Django, Lift, Pylons, Node.js-based systems, etc.

Design Goals

The design goals and constraints of PelletServer include:

  • Integration. The power of Semantic Web technologies lies primarily in their ability to accelerate information integration and analysis. And web technologies are already the basis of most integration efforts. PelletServer blends the best semantic and web technologies into a single, coherent framework.
  • Universal Access. Most Semantic Web software is built with Java. But what about Ruby, Python, JavaScript, C#, etc? Since HTTP is the universal access layer, PelletServer provides programmatic access to its services via RESTful HTTP.
  • Simplicity. We've applied conventions and sane defaults aggressively in the design and implementation of PelletServer; most services are accessed as easily as 
    $ wget http://example.com/kb/wine/consistency
  • Standards Compliance. We built PelletServer using standards and standard-compliant components, including HTTP, SPARQL, OWL, RDF, AMQP, and JSON.
  • Performance. Since many of our commercial products are based on computationally complex algorithms, we can't afford to ignore performance. So we designed PelletServer to be multi-machine, distributed, asynchronous, and cloud-friendly from the beginning. PelletServer uses AMQPBoth asynchronous HTTP interaction and distributed AMQP services are scheduled for PelletServer 1.0 release. internally to manage communications between distributed systems. The HTTP interface uses asynchronous patterns of client-server interaction when requests can't be computed in a few seconds.
  • URL Agnosticism. Some people naturally think of HTTP-based information integration from the resource-centric perspective. In that view, URLs identify resources, i.e., primarily data resources. Other people naturally think of URLs as identifying services, i.e., computational artifacts which take data as arguments. We think both perspectives are valuable; while PelletServer's out-of-the-box URL design is resource-centric, we provide a mechanism to rewrite PelletServer URLs dynamically in either style.In service-centric style, the example above looks like 
    $ wget http://www.example.com/consistency/wine
